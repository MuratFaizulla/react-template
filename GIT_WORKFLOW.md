## 1. Что такое ветка (branch)

**Ветка** — это отдельная копия кода, в которой ты можешь делать изменения, не затрагивая основной код.

Представь папку с проектом. Ветка — это как создать копию этой папки, поработать в ней, и потом решить — добавить изменения в оригинал или выбросить.

```
main (основная ветка)
│
├── feat/login-page     ← ты работаешь тут
├── fix/auth-bug        ← или тут
└── docs/update-readme  ← или тут
```

**Главное правило:** ветка `main` — это всегда рабочий, готовый код. Никогда не ломай `main`.

## 2. Что такое PR (Pull Request)

**PR (Pull Request)** — это запрос на добавление твоих изменений из ветки в `main`.

### Как это выглядит на практике:

1. Ты создал ветку `feat/user-profile`
2. Сделал изменения, закоммитил
3. Запушил ветку на GitHub
4. Открыл PR: "хочу добавить мой код из `feat/user-profile` в `main`"
5. GitHub показывает все изменения — что добавил, что удалил
6. Ты (или другой разработчик) проверяет изменения
7. Нажимаешь **Merge** — код попадает в `main`

### Зачем это нужно:

- **Безопасность** — плохой код не попадёт в `main` случайно
- **История** — каждое изменение зафиксировано как отдельный PR
- **Проверка** — можно посмотреть что изменилось перед добавлением
- **Откат** — если что-то сломалось, легко найти какой PR виноват

### Как выглядит PR на GitHub:

```
┌─────────────────────────────────────────────┐
│  feat: add user profile page        #4      │
│                                             │
│  feat/user-profile → main                   │
│                                             │
│  +85 строк   -12 строк   3 файла            │
│                                             │
│  [Merge pull request]                       │
└─────────────────────────────────────────────┘
```

## 3. Как устроена защита main

В этом проекте настроены два уровня защиты `main`:

### Уровень 1 — Локальный хук (твой компьютер)

Файл `.git/hooks/pre-push` блокирует прямой пуш в `main`.
Если попробуешь сделать `git push origin main` — получишь ошибку:

```
Direct push to 'main' is not allowed.
Create a feature branch and open a PR:

  git checkout -b fix/your-branch-name
  git push origin fix/your-branch-name
```

### Уровень 2 — GitHub Branch Protection (сервер)

Даже если обойдёшь хук, GitHub не примет прямой пуш в `main`.
Настроено в: **Settings → Branches → main**

## 4. Полный процесс работы шаг за шагом

### Шаг 1 — Убедись что ты на main и он актуален

```bash
git co main        # переключись на main
git pl             # подтяни последние изменения с GitHub
```

Это важно делать перед каждой новой задачей, чтобы твоя ветка создалась от актуального кода.

### Шаг 2 — Создай новую ветку

```bash
git co -b feat/название-твоей-задачи
```

Примеры:
```bash
git co -b feat/user-profile       # новая фича
git co -b fix/login-redirect-bug  # исправление бага
git co -b docs/add-api-guide      # документация
git co -b refactor/auth-flow      # рефакторинг
```

### Шаг 3 — Делай изменения в коде

Открываешь файлы в редакторе, редактируешь, сохраняешь.

Периодически проверяй что изменилось:
```bash
git st    # покажет какие файлы изменены
git df    # покажет конкретные изменения в коде
```

### Шаг 4 — Добавь изменения в коммит

```bash
# Добавить конкретный файл:
git ad src/pages/LoginPage/LoginPage.tsx

# Добавить несколько файлов:
git ad src/pages/LoginPage/LoginPage.tsx src/pages/LoginPage/LoginPage.module.css

# Добавить все изменённые файлы:
git ad .
```

> ⚠️ Будь осторожен с `git ad .` — он добавляет ВСЕ файлы, включая случайно изменённые.

### Шаг 5 — Создай коммит

```bash
git cm -m "feat: добавил страницу профиля пользователя"
```

О том как правильно писать сообщения коммитов — раздел [6. Правила написания коммитов](#6-правила-написания-коммитов).

### Шаг 6 — Повторяй шаги 3–5 пока задача не готова

Можно делать несколько коммитов на одну задачу:

```bash
git cm -m "feat: добавил компонент ProfileCard"
git cm -m "feat: добавил стили для ProfileCard"
git cm -m "fix: исправил отображение аватара"
```

### Шаг 7 — Запушь ветку на GitHub

```bash
git ps origin feat/user-profile
```

GitHub выведет ссылку:
```
remote: Create a pull request by visiting:
remote: https://github.com/MuratFaizulla/react-template/pull/new/feat/user-profile
```

### Шаг 8 — Создай и смержи PR

Можешь попросить меня:
> "закоммить и смержи PR"

Я автоматически создам PR через GitHub API и смержу его.

Или сделай сам на GitHub — открой ссылку из шага 7 и нажми **Create pull request**, потом **Merge pull request**.

### Шаг 9 — Подтяни обновлённый main

```bash
git co main    # вернись на main
git pl         # подтяни изменения (твой код теперь тут)
```

### Шаг 10 — Удали ветку

```bash
git branch -D feat/user-profile             # удалить локально
git push origin --delete feat/user-profile  # удалить на GitHub
```

## 5. Именование веток

Ветки называются по шаблону: `тип/краткое-описание`

| Тип | Когда использовать | Пример |
|---|---|---|
| `feat/` | Новая функция или страница | `feat/user-profile` |
| `fix/` | Исправление бага | `fix/login-redirect` |
| `refactor/` | Переработка кода без изменения поведения | `refactor/auth-hooks` |
| `chore/` | Настройки, зависимости, конфиг | `chore/update-eslint` |
| `docs/` | Документация | `docs/api-guide` |
| `style/` | Только стили, без логики | `style/button-hover` |

**Правила:**
- Только строчные буквы
- Слова разделяются через `-`
- Коротко и понятно
- На английском

## 6. Правила написания коммитов

Коммиты пишутся по формату **Conventional Commits**:

```
тип: краткое описание что сделал
```

| Тип | Когда использовать |
|---|---|
| `feat` | Новая функция |
| `fix` | Исправление бага |
| `refactor` | Рефакторинг |
| `style` | Форматирование, стили |
| `chore` | Конфиг, зависимости |
| `docs` | Документация |
| `perf` | Оптимизация производительности |
| `revert` | Откат предыдущего коммита |

### Примеры хороших коммитов:

```bash
git cm -m "feat: add user profile page"
git cm -m "fix: resolve infinite loop in useQuery"
git cm -m "refactor: extract validation logic to separate hook"
git cm -m "chore: update vite to v7"
git cm -m "docs: add git workflow guide"
git cm -m "style: fix button hover animation"
```

### Примеры плохих коммитов (так не надо):

```bash
git cm -m "изменения"          # непонятно что сделано
git cm -m "fix"                # слишком коротко
git cm -m "работает!!!"        # эмоции вместо описания
git cm -m "wip"                # work in progress — не коммить сырой код
```

**Правила:**
- Пиши на английском
- После типа ставь двоеточие и пробел
- С маленькой буквы
- Без точки в конце
- Коротко (до 72 символов)

## 7. Короткие команды (алиасы)

Вместо длинных команд используй короткие алиасы:

| Алиас | Полная команда | Что делает |
|---|---|---|
| `git ad` | `git add` | Добавить файлы в коммит |
| `git cm -m "..."` | `git commit -m "..."` | Создать коммит |
| `git st` | `git status` | Показать что изменено |
| `git df` | `git diff` | Показать изменения в коде |
| `git ps` | `git push` | Отправить на GitHub |
| `git pl` | `git pull` | Подтянуть с GitHub |
| `git co` | `git checkout` | Переключиться на ветку |
| `git br` | `git branch` | Список веток |
| `git lg` | `git log --oneline --graph --decorate --all` | Красивая история коммитов |
| `git sh` | `git stash` | Временно спрятать изменения |
| `git rb` | `git rebase` | Перебазировать ветку |

## 8. Настройки Git

В глобальный конфиг (`~/.gitconfig`) настроены три важные опции:

### `pull.rebase = true`

```bash
# Обычный git pull (было):
# main: A - B - C - M (merge commit)  ← некрасиво
#              ↘   ↗
#               D

# С rebase (стало):
# main: A - B - C - D  ← чистая линейная история
```

Вместо создания лишнего "merge commit" при `git pull`, твои коммиты просто переносятся поверх новых изменений. История остаётся чистой и линейной.

### `rebase.autoStash = true`

```bash
# Без autoStash — ошибка:
git pl   # error: cannot pull with rebase: you have unstaged changes

# С autoStash — работает автоматически:
git pl   # 1. Прячет твои изменения (stash)
         # 2. Подтягивает новый код
         # 3. Восстанавливает твои изменения
```

Если у тебя есть несохранённые изменения и ты делаешь `git pull` — Git автоматически их спрячет, подтянет код, и вернёт твои изменения обратно.

### `rerere.enabled = true`

```bash
# Ты решил конфликт в файле auth.ts
# Git запомнил решение

# Та же ветка снова конфликтует (например после rebase)
# Git автоматически применяет то же решение — не нужно решать снова
```

`rerere` = **Re**use **Re**corded **Re**solution. Запоминает как ты решал конфликты и применяет эти решения автоматически в похожих ситуациях.

## 9. Частые ситуации и как их решать

### "Случайно начал работать в main"

```bash
# Создай ветку прямо сейчас — коммиты сохранятся
git co -b feat/my-feature

# Сброс main до origin/main
git br -f main origin/main
```

### "Нужно переключиться на другую ветку, но есть несохранённые изменения"

```bash
# Спрячь изменения
git sh

# Переключись на другую ветку
git co main

# ... сделал что нужно ...

# Вернись и восстанови изменения
git co feat/my-feature
git sh pop
```

### "Хочу посмотреть историю коммитов"

```bash
git lg
```

Вывод:
```
* ff31848 (HEAD -> main) fix: replace $TSFixMe with proper types
* 59260c5 docs: update README
* 9841866 chore: add CLAUDE.md
* 7d6c200 fix: resolve all critical bugs
```

### "Нужно отменить последний коммит (не запушенный)"

```bash
# Отменить коммит, но оставить изменения в файлах
git reset HEAD~1

# Отменить коммит и выбросить изменения (осторожно!)
git reset --hard HEAD~1
```

### "Ветка отстала от main — нужно обновить"

```bash
git co feat/my-feature   # перейди на свою ветку
git rb main              # перебазируй поверх main
```

### "Конфликт при rebase"

```bash
# Git скажет в каком файле конфликт
# Открой файл, найди конфликт:

<<<<<<< HEAD
  твой код
=======
  код из main
>>>>>>> main

# Выбери нужный вариант, удали маркеры конфликта
# Затем:
git ad .
git rb --continue
```

## Краткая шпаргалка

```bash
# Начало новой задачи
git co main && git pl
git co -b feat/my-feature

# Работа
git st                          # что изменено?
git df                          # как изменено?
git ad src/path/to/file.tsx     # добавить файл
git cm -m "feat: add something" # коммит

# Отправка
git ps origin feat/my-feature   # запушить ветку

# После merge PR
git co main && git pl
git branch -D feat/my-feature
git push origin --delete feat/my-feature
```
